# Аудит безопасности и соответствия ТЗ ShoShop

## Выполненные исправления (реальная защита)

### 1. Callback платежей
- **Проблема:** Любой мог отправить POST на `/api/payments/callback/` с `order_id` и `status=success` и получить бесплатную выдачу товара.
- **Исправлено:**
  - Проверка секрета: заголовок `X-Payment-Secret` или поле `callback_secret`/`secret` в body. Секрет задаётся в `PAYMENT_CALLBACK_SECRET` (в `.env`). Без секрета в настройках проверка пропускается (только для dev/stub).
  - Поддержка JSON body (многие платёжные шлюзы шлют `application/json`).
  - Обработка только заказов в статусе `PENDING`; несуществующий или уже оплаченный заказ игнорируется.
  - Идемпотентность: повторный callback по уже оплаченному заказу не дублирует выдачу товаров.

### 2. Идемпотентность оплаты и выдачи
- В `apply_payment_result`: если заказ уже `PAID` или не в `PENDING`/`CREATED`, функция ничего не делает.
- Исключена двойная выдача цифровых товаров при повторном вызове callback.

### 3. Корзина и заказы
- Ограничение количества одного товара в корзине: не более 99 (`CART_MAX_QTY`).
- Валидация при оформлении заказа: только положительные `product_id` и `qty` в допустимом диапазоне; некорректные записи отбрасываются.
- В `create_order_for_user`: проверка существования товара и статуса `active`, валидация `qty` (1–99). При пустом списке товаров выбрасывается `ValueError` (редирект обратно в корзину).

### 4. Rate limit входа, регистрации и сброса пароля
- При превышении лимита попыток логина возвращается редирект на страницу входа с сообщением об ошибке (вместо JSON 429).
- Те же лимиты (по IP + идентификатору) применены к **регистрации** и **запросу сброса пароля**, чтобы снизить риск спама и перебора.

### 5. Заголовки безопасности
- **SecurityHeadersMiddleware** добавляет к каждому ответу: `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` (ограничение геолокации/микрофона/камеры), `X-XSS-Protection: 1; mode=block`.

### 6. Настройки безопасности
- **DEBUG** больше не передаётся в шаблоны через context processor (снижение риска утечки в проде).
- **HSTS:** при `DEBUG=True` отправка HSTS отключена (`SECURE_HSTS_SECONDS = 0`), чтобы не ломать локальную разработку.

## Уже реализовано (соответствие лучшим практикам)

- **CSRF:** включён глобально; callback платежей освобождён по необходимости, доступ защищён секретом.
- **Сессии:** `SESSION_COOKIE_HTTPONLY`, `SESSION_COOKIE_SECURE` в проде, `SESSION_COOKIE_SAMESITE=Lax`, `CSRF_COOKIE_SAMESITE=Lax`, `CSRF_COOKIE_HTTPONLY`.
- **Пароли:** валидаторы Django (длина, сложность, типичные пароли); хранение через встроенный хешер.
- **Доступ к данным:** заказы и API заказов отфильтрованы по `request.user`; админка — только для staff.
- **Логирование безопасности:** неуспешный вход пишется в `SecurityEvent`; в админке есть журнал безопасности.
- **Ограничение попыток входа:** по IP и имени пользователя, окно и лимит настраиваются через `LOGIN_RATE_LIMIT` и `LOGIN_RATE_WINDOW`.

## Рекомендации для продакшена

1. **Секрет callback:** задать `PAYMENT_CALLBACK_SECRET` в `.env` и настроить платёжный шлюз на отправку этого секрета (заголовок `X-Payment-Secret` или поле в body).
2. **DEBUG:** убедиться, что `DJANGO_DEBUG=false` и `DJANGO_SECRET_KEY` уникальный и не попал в репозиторий.
3. **ALLOWED_HOSTS:** задать явный список доменов.
4. **HTTPS:** включить `DJANGO_SECURE_SSL_REDIRECT=true` за обратным прокси с SSL.
5. **Заголовки:** приложение уже добавляет X-Content-Type-Options, Referrer-Policy, Permissions-Policy; X-Frame-Options даёт Django. При необходимости добавить CSP (Content-Security-Policy) в nginx или middleware.
6. **Резервное копирование БД** и тестовые восстановления по расписанию.
7. **Чек-лист перед релизом:** см. также **ROADMAP_IMPROVEMENTS.md** (раздел про чек-лист и env.example).

## Визуальные и технические нюансы (не критичные)

- Единый стиль карточек и кнопок на главной и в каталоге — приведён к одному виду.
- Адаптив: на малых экранах навигация скрывается, поиск уходит в отдельную строку — при необходимости можно добавить бургер-меню.
- Поиск в шапке: подсказки через API; при отсутствии результатов можно явно показывать «Ничего не найдено».
- Страницы 404/500: кастомные шаблоны есть; при желании можно усилить контент и стили.
